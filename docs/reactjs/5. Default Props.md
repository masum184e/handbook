## Why do we need Default Props and PropTypes?

When working with props, problems commonly happen when:

- A prop is missing
- A prop has the wrong type
- A component assumes data that isn’t guaranteed

Default props give fallback values  
PropTypes validate props during development

## What are default props?

Default props are values used when a prop is:

- Not passed
- Passed as undefined

This ensures the component still works.


## Example without default props (problem)
```jsx
function Greeting({ name }) {
  return <h2>Hello, {name}</h2>;
}
```
If used like this:
```jsx
<Greeting />
```
Output:
```
Hello, undefined
```
## Using default props (modern approach)

With function components, we usually use default parameters.
```jsx
function Greeting({ name = "Guest" }) {
  return <h2>Hello, {name}</h2>;
}
```
- If `name` is missing → `"Guest"` is used
- If `name` is provided → it overrides the default

## Usage
```
<Greeting />
<Greeting name="Alice" />
```
Outputs:
```
Hello, Guest
Hello, Alice
```
## Default props with multiple values
```jsx
function Button({ type = "button", disabled = false }) {
  return (
    <button type={type} disabled={disabled}>
      Click Me
    </button>
  );
}
```

This prevents:

- Missing attributes
- Unexpected behavior

## Default props vs state

Important distinction:

| Feature               | Default Props | State |
| --------------------- | ------------- | ----- |
| Can change internally | ❌             | ✅     |
| Provided by parent    | ✅             | ❌     |
| Used as fallback      | ✅             | ❌     |
| Triggers re-render    | ❌             | ✅     |

Don’t use default props to manage dynamic data — that’s state’s job.

## What is PropTypes

PropTypes is a library (`prop-types`) used to:

- Validate props at runtime (development only)
- Warn if props have the wrong type or shape

It does not:

- Stop the app
- Affect production builds


## Basic PropTypes example
```
import PropTypes from "prop-types";

function User({ name, age }) {
  return (
    <p>
      {name} is {age} years old
    </p>
  );
}
```

Adding PropTypes
```
User.propTypes = {
  name: PropTypes.string,
  age: PropTypes.number,
};
```
**What happens?**

If someone passes:
```jsx
<User name={25} age="twenty" />
```

React shows warnings in the console:
```
Warning: Failed prop type: Invalid prop `name` of type `number` supplied to `User`
```
## Required props

Use `.isRequired` when a prop must be provided.

```jsx
User.propTypes = {
  name: PropTypes.string.isRequired,
  age: PropTypes.number,
};
```

If `name` is missing:
```
Warning: The prop `name` is marked as required but its value is undefined
```
## PropTypes for complex data (objects & arrays)
Object shape
```jsx
User.propTypes = {
  user: PropTypes.shape({
    name: PropTypes.string.isRequired,
    email: PropTypes.string,
    age: PropTypes.number,
  }),
};
```
Array of values
```jsx
List.propTypes = {
  items: PropTypes.arrayOf(PropTypes.string),
};
```
Array of objects
```jsx
Posts.propTypes = {
  posts: PropTypes.arrayOf(
    PropTypes.shape({
      id: PropTypes.number.isRequired,
      title: PropTypes.string.isRequired,
    })
  ),
};
```
## Default props + PropTypes together (best practice)
```jsx
function Avatar({ image, size }) {
  return <img src={image} width={size} height={size} />;
}

Avatar.defaultProps = {
  size: 50,
};

Avatar.propTypes = {
  image: PropTypes.string.isRequired,
  size: PropTypes.number,
};
```
- `size` defaults to `50`
- `image` must be provided
- Warnings appear if rules are broken


PropTypes only warn — they don’t replace checks.

## PropTypes vs TypeScript
| Feature             | PropTypes | TypeScript |
| ------------------- | --------- | ---------- |
| Runtime checking    | ✅         | ❌          |
| Compile-time safety | ❌         | ✅          |
| Learning curve      | Low       | Higher     |
| Production impact   | None      | None       |
