(a) Project Scope for the OVS Scenario

Definition​

The Project Scope defines the total volume of work and specific deliverables agreed upon for the Online Video Store (OVS) system.

Scope Includes:​

  • Video Database Management

    • System to catalogue videos and maintain a searchable video database.

  • Customer Registration System

    • Module to register customers and validate OVS customer cards.

  • Rental Transaction Tracking

    • Manage three-day rental cycles and return processing.

  • Automated Compliance Checks

    • Automatically block rentals if customers have overdue videos or unpaid fees.

  • Reporting Tools

    • Generate daily overdue reports for the store manager.

  • Inventory Maintenance

    • Remove damaged videos from the database.
    • Calculate and apply customer charges when necessary.
(b) Project Objectives for the OVS Scenario

Project Objectives (High-Level Goals)​

  • Operational Efficiency

    • Reduce manual errors in rental tracking and eligibility verification.

  • Revenue Protection

    • Ensure 100% collection of overdue fees.
    • Maintain accountability for damaged inventory.

  • Improved Customer Service

    • Streamline registration and rental processes.
    • Reduce verification delays for valid customers.

  • Effective Resource Management

    • Provide timely overdue reports for proactive follow-up by the manager.
(c) Difficulties Faced During Project Initiation

Software projects commonly face the following challenges:

  • Unclear Vision

    • Stakeholders may not clearly define system requirements.

  • Lack of Visibility

    • Difficulty in estimating cost, effort, and schedule with immature requirements.

  • Misunderstanding

    • Gaps between stakeholder expectations and development team interpretation.

  • High Risk of Failure

    • Poorly defined charter and scope may fail to secure management support.
(d) Ensuring Well-Defined Project Scope

To prevent scope creep, the following steps should be taken:

  • Identify Detailed Deliverables

    • Clearly describe each deliverable to eliminate ambiguity.

  • Freeze Quality Levels

    • Define reliability, security, and performance standards early.

  • Establish Change Request Mechanism

    • Analyze cost and schedule impact before approving requirement changes.

  • Align Stakeholder Expectations

    • Ensure practical alignment between business goals and technical feasibility.
(e) Factors That Make Project Management Vary

Project management differs depending on:

  • Technology and Platform

    • Web-based systems vs. mainframe applications require different approaches.

  • Project Size and Complexity

    • Large projects require structured methodologies.
    • Small projects may use informal methods.

  • Team Distribution

    • Distributed teams require stronger communication and coordination frameworks.

  • Development Model

    • Waterfall vs. Agile depends on requirement clarity and contract type.

  • Organizational Environment

    • Company policies and culture influence management style.

(a) Draw the PERT chart for the above tasks

The PERT (Program Evaluation and Review Technique) chart represents the network of tasks and their dependencies.

Logical Flow Breakdown:

  • Sequential Start: Task A leads to Task B.
  • Branching: Task B branches into Task C and Task D.
  • Convergence: Task F requires both C and D. Task G requires D and F. Task I requires E and F.
  • Final Bottle-neck: Task J cannot begin until G, H, and I are all completed.
  • Conclusion: Tasks J, K, and L are strictly sequential to finish the project.
(b) Find the Critical Path and Minimum Duration

To find the critical path, we calculate the Earliest Finish (EF) for every task. The EF of a task is the maximum EF of its predecessors plus its own duration.

Steps to Identify the Path:

  1. Analyze Task J's start time: It depends on G (37), H (39), and I (38). Since it must wait for the longest one, Task H determines the start of Task J.
  2. Trace Backward: Following the longest durations back to the start: L ← K ← J ← H ← F ← C ← B ← A.

Results:

  • Critical Path: A → B → C → F → H → J → K → L
  • Minimum Duration: 56 Days
(c) Impact of Task H taking 2 days longer

The Effect: The project finishing date will be delayed by exactly 2 days, resulting in a total duration of 58 days.

The "Why":

  • Critical Path Membership: Task H is a "Critical Task." By definition, tasks on the critical path have zero float (slack).
  • The Bottleneck Effect: Task J (System Testing) can only begin once G, H, and I are all finished. Currently, H is the latest to finish (Day 39). If H takes 9 days instead of 7, it finishes on Day 41.
  • Mathematical Impact: Since 41>3841 > 3841>38 (Task I) and 41>3741 > 3741>37 (Task G), the start date for all subsequent tasks (J, K, and L) is pushed back by 2 days.

Question 6: Software Project Management Basics​

6. (a) What is software project management? Why is software project management important?

Definition​

Software Project Management (SPM) is a proper way of planning, monitoring, and controlling software projects. It involves the use of knowledge, skills, tools, and techniques to meet project requirements within specific constraints like time, budget, and quality.

Importance of SPM​

SPM is crucial because software development is an abstract process with high complexity. Its importance includes:

  • Managing Scope: Prevents "scope creep" by ensuring only agreed-upon features are developed.
  • Resource Optimization: Ensures that developers, testers, and infrastructure are used efficiently.
  • Risk Mitigation: Identifies potential bottlenecks (like technical debt or staff turnover) before they derail the project.
  • Quality Assurance: Ensures the final product meets the client's standards and functions reliably.
  • Cost & Time Control: Keeps the project within the financial budget and ensures timely delivery to the market.
(b) What key responsibilities will you have as a software project manager to ensure the project's success (E-commerce case study)?

Given the tight deadlines and evolving requirements of the e-commerce platform, your key responsibilities include:

  1. Defining the Project Lifecycle: Choosing an Agile/Scrum methodology to handle "evolving requirements" through iterative sprints.
  2. Stakeholder Management: Acting as a bridge between the retail client, the development team, and end-users to manage expectations and gather feedback.
  3. Task Scheduling & Prioritization: Using tools (like Jira or Gantt charts) to ensure high-priority features (like the payment gateway) are built first.
  4. Risk Management: Identifying risks early—such as security vulnerabilities in the payment system—and creating contingency plans.
  5. Team Leadership & Motivation: Clearing "blockers" for the developers to maintain high velocity under tight deadlines.
(c) Why are discounted cash flow (DCF) techniques considered more effective for project selection than using net profit or ROI?

Discounted Cash Flow (DCF) is superior because it accounts for the Time Value of Money (TVM)—the principle that a dollar today is worth more than a dollar tomorrow.

FeatureNet Profit / ROIDiscounted Cash Flow (DCF)
Time ValueIgnores when the money is earned.Accounts for the timing of cash flows.
Risk AssessmentTreats all future income as guaranteed.Uses a "discount rate" to account for future uncertainty and inflation.
AccuracySimple but can be misleading for long-term projects.Provides a more realistic "Net Present Value" (NPV).

Summary: While ROI tells you how much you make, DCF tells you what that future profit is actually worth in today’s terms, allowing for a more "apples-to-apples" comparison between different projects.

Question 7: Project Failure and Estimation​

(a) Why do software development projects fail? Explain with proper example/s.

Software projects often fail due to a combination of technical and managerial issues:

  1. Poor Requirement Analysis: Requirements are often vague or misunderstood.
  • Example: A client asks for a "search bar," but the developers build a simple text filter that can't handle the database load of 1 million products.
  1. Unrealistic Deadlines: Pressure from stakeholders leads to "death marches."
  • Example: Forcing a 6-month project into a 2-month window, resulting in buggy, unmaintainable code.
  1. Lack of User Involvement: Building a product that the end-user doesn't actually find useful.
  2. Technical Debt: Using outdated technology or rushing code, leading to a system that eventually crashes or cannot be updated.
(b) What remedial measures can be taken so that software development projects do not fail?

To prevent failure, project managers should implement:

  • Iterative Development (Agile): Deliver small pieces of working software frequently to get immediate feedback.
  • Rigorous Testing: Implement Automated Testing and Continuous Integration (CI/CD) to catch bugs early.
  • Clear Documentation: Ensure requirements and system architecture are well-documented.
  • Effective Communication: Hold daily stand-ups to identify and solve problems immediately.
  • Buffer Management: Always include a "buffer" in the schedule to account for unforeseen technical challenges.
(c) Describe the COCOMO technique for deriving effort and cost estimates for software projects.

The Constructive Cost Model (COCOMO) is an algorithmic model used to estimate the cost, effort, and schedule of a project based on the size of the software.

1. Project Types​

COCOMO categorizes projects into three types:

  • Organic: Small teams, familiar environment (e.g., a simple payroll system).
  • Semi-Detached: Intermediate complexity and team size (e.g., a new database system).
  • Embedded: High complexity, tight constraints (e.g., air traffic control software).

2. The Levels​

  • Basic COCOMO: Uses a simple formula based on Lines of Code (KLOC).
  • Intermediate COCOMO: Applies "Cost Drivers" (Product reliability, hardware constraints, personnel capability).
  • Detailed COCOMO: Accounts for the influence of individual project phases (analysis, design, etc.).

3. The Formula (Basic)​

The effort EEE is typically calculated as:

E=a⋅(KLOC)bE = a \cdot (KLOC)^bE=a⋅(KLOC)b

Where a and b are constants based on the project type.
