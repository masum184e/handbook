### Review Questions

<details>
<summary><strong>2.1 What is a project charter? How can you ensure that the project charter is useful for the project?</strong></summary>

### What is a Project Charter?

A **project charter** is a high-level document issued by top management that captures:

- The **stakeholders' vision**
- The overall **big picture**
- Information that may remain constant across incremental projects

### How to Ensure It Is Useful

To make the project charter effective:

- Clearly define **project goals**
- Specify **project objectives**
- Identify **major responsibility allocations**
- Link the software project to broader **business goals**

</details>

<details>
<summary><strong>2.2 What things should go on the list containing project objectives?</strong></summary>

### Guidelines

- Use **clear language**
- Keep the list **concise and focused**

### Typical Items

- **Operational Goals**
  - Example: Reduce or eliminate paper-based transactions to cut processing time.

- **Business Gains**
  - Example: Centralize functions to reduce costs.
  - Increase customer satisfaction percentage.

- **Success Criteria**
  - Clearly defined tasks explaining _how_ the charter will be fulfilled.
  - Example: Add validation checks to ensure 100% process accuracy.

</details>

<details>
<summary><strong>2.3 How can you ensure that the project scope is well defined at initiation so it does not get overstretched later?</strong></summary>

### Key Steps

- **Identify Detailed Deliverables**
  - Clearly describe every deliverable.
  - Remove ambiguity.

- **Define Quality Levels**
  - Freeze required quality attributes early (reliability, security, performance).
  - Quality level directly affects total workload.

- **Establish Change Management**
  - Define a formal **change request process**.
  - Communicate that requirement changes impact cost and schedule.

</details>

<details>
<summary><strong>2.4 What difficulties are faced during project initiation?</strong></summary>

### Common Challenges

- **Unclear Vision**
  - Stakeholders may know they need a system but cannot clearly define it.

- **Lack of Visibility**
  - Difficult to estimate effort, cost, and timeline with immature requirements.

- **High Risk**
  - Greatest stakeholder influence occurs here.
  - Lack of strong top management support may terminate the project early.

</details>

<details>
<summary><strong>2.5 What is the relation between quality level and project scope?</strong></summary>

### Relationship Explained

- Quality level is an **integral part of project scope**.
- Total work volume depends on:
  - Number of features
  - Required quality level

### Example

- A **surveillance system** requires high reliability and security → Higher effort.
- A simple **game application** → Lower quality demands → Lower effort.

### Conclusion

The scope document must clearly define quality expectations to determine:

- Accurate budget
- Realistic schedule

</details>

<details>
<summary><strong>2.6 What other activities are performed during project initiation?</strong></summary>

### Additional Initiation Activities

- **Initial Estimation**
  - Rough schedule
  - Effort estimation
  - Cost estimation

- **Risk Identification**
  - Identify potential risks
  - Develop initial mitigation strategies

- **Feasibility Study**
  - Conduct a “make or break” analysis
  - Assess technical and financial viability

- **Project Division**
  - Decide whether to split into:
    - Requirement-clearance phase
    - Development phase

- **Quality Planning**
  - Define framework for measuring and ensuring quality

</details>
