## Types of Software Changes/Maintainance

In software engineering, these four categories (often referred to as **Swanson’s Maintenance Categories**) define the entire lifecycle of a product after its initial release. Understanding the nuances between them helps in resource planning and technical debt management.

![Software Changes](/img/software-maintainance/relation_between_software_changes.png)

### Corrective Maintenance

This is "reactive" maintenance. It involves identifying, isolating, and fixing bugs or faults that were not discovered during the initial testing phase but surfaced during actual use.

- **Rationale:** To restore the software to its specified functional state.
- **Example:** A banking app crashes whenever a user tries to transfer exactly $10,000 because of a "buffer overflow" error in the code.
- **Explanation:** The developer must find the specific line of code causing the crash, fix the logic, and push a patch. This is corrective because it fixes a **failure** to meet existing requirements.

### Adaptive Maintenance

This involves modifying the software to keep it compatible with a changing **environment**. The "environment" includes the operating system, hardware, cloud platforms, or third-party APIs the software relies on.

- **Rationale:** To ensure the software does not become obsolete due to external changes.
- **Example:** A retail website must be updated because Google Chrome released a new version that blocks the specific way the site currently handles "cookies."
- **Explanation:** The website’s features were working perfectly, but the **external environment** (the browser) changed. The developer must adapt the code to comply with the new browser standards to keep the site functional.

### Perfective Maintenance

This is "proactive" maintenance aimed at enhancing the software. It involves adding new features, improving the user interface (UI), or optimizing code to make it run faster or use less memory.

- **Rationale:** To improve the system's value and meet evolving user expectations.
- **Example:** A video editing software adds a new "AI Background Remover" tool because users requested it, and it improves the processing speed of exports by 20%.
- **Explanation:** This isn't fixing a bug or reacting to an OS update; it is **perfecting** the product by adding new utility and efficiency to make it "better" than the original version.

### Preventive Maintenance (Software Re-engineering)

This involves making changes to the software to prevent **future** problems. It often includes refactoring code, updating documentation, or cleaning up "spaghetti code" to make the system easier to maintain later.

- **Rationale:** To reduce "technical debt" and increase the long-term maintainability of the system.
- **Example:** A developer notices that a specific module is very cluttered and hard to read. Even though it works fine now, they spend two days rewriting it into a cleaner, modular structure.
- **Explanation:** This change is invisible to the end-user. However, by cleaning the code now, the developer **prevents** future bugs and ensures that when the time comes to add a new feature, it won't take twice as long due to messy code.

## Incremental Release
An **Incremental Release** is a strategy where software updates are delivered in small, manageable "chunks" or versions rather than waiting to release one massive overhaul.

* **How it works:** Instead of releasing "Version 2.0" after a year of work, the maintenance team releases Version 1.1 (a few bug fixes), then 1.2 (one new feature), then 1.3 (a security patch).
* **Benefits in Maintenance:**
    * **Reduced Risk:** If a bug is introduced, it is easier to identify which small change caused it.
    * **Faster Feedback:** Users get improvements sooner and can tell developers if the changes meet their needs.
    * **Stability:** It follows **Lehman’s Law of Familiarity**, ensuring users and developers aren't overwhelmed by too many changes at once.

## The Ripple Effect

The **Ripple Effect** is a phenomenon where a small change in one part of the software unintentionally causes errors or requires further changes in other, seemingly unrelated parts of the system.

* **The Cause:** This usually happens due to high **coupling** (where different code modules are tightly "glued" together). 
* **The Problem:** It is the primary reason why maintenance is expensive. A developer might fix a bug in the "Login" module, but that fix "ripples" out and breaks the "Payment" module because they shared a specific variable.
* **How to Manage It:**
    * **Impact Analysis:** Before making a change, developers study the code to see what else it touches.
    * **Regression Testing:** Running a full suite of tests after *every* change to catch these ripples before the user sees them.
* **Example:** Changing a date format from `MM/DD/YY` to `DD/MM/YYYY` in a database. If the "Report Generator" isn't updated to handle the extra digits, it will crash.

## Lehman Laws

M Meir "Manny" Lehman, a researcher at IBM, formulated these laws based on his studies of how large software systems evolve. Unlike the laws of physics, these are **observations of industrial behavior**—they describe what _actually_ happens over time as software is used and maintained.

The laws are generally categorized into eight principles, but the first five are the most widely cited in software engineering.

### 1. Law of Continuing Change (E-type systems)

- **The Concept:** A software system that is used in a real-world environment (an "E-type" system) **must** continually change, or it becomes progressively less useful.
- **Why:** The world around the software changes (new laws, new competitors, new user habits), so the software must "evolve" to stay relevant.
- **Example:** A tax filing app from 2020 is useless in 2026 unless it is updated with the latest tax codes.

### 2. Law of Increasing Complexity

- **The Concept:** As a system evolves, its **complexity increases** unless specific work is done to maintain or reduce it.
- **Why:** Every time you add a new "quick fix" or a new feature, you create more interconnections in the code. Without refactoring, the system eventually becomes a "spaghetti" mess.
- **Example:** Adding 50 new buttons to a simple interface over five years eventually makes the underlying code so tangled that even small changes become risky.

### 3. Law of Self-Regulation

- **The Concept:** Global system evolution is a **self-regulating process**. Product attributes like size, time between releases, and the number of reported errors remain statistically stable over time.
- **Why:** Large organizations have a natural "cadence." If a team tries to rush a huge update, they will likely be slowed down by a massive wave of bugs in the next cycle, bringing the average progress back to normal.

### 4. Law of Conservation of Organizational Stability (Invariant Work Rate)

- **The Concept:** The average effective work rate on a software system is **constant** over time, regardless of the resources pushed into it.
- **Why:** Adding more programmers to a late project often makes it later (Brooks's Law). The overhead of communication and the complexity of the code act as a "speed limit" on how much work can actually be finished.

### 5. Law of Conservation of Familiarity

- **The Concept:** As a system grows, the amount of new content in each release must be limited to ensure that stakeholders (developers and users) can remain **familiar** with the system.
- **Why:** If a release contains too many changes, the developers will lose their "mental map" of how the system works, leading to a spike in errors.

### The Remaining Three Laws (Briefly)

6.  **Continuing Growth:** The functional content of a system must continually increase to maintain user satisfaction.
7.  **Declining Quality:** The quality of a system will appear to decline unless it is rigorously maintained and adapted to operational environment changes.
8.  **Feedback System:** Evolution processes are multi-level, multi-loop, multi-agent feedback systems and must be treated as such to be managed effectively.

### Summary Table for Quick Reference

| Law                | Short Version                  | Management Impact                                     |
| :----------------- | :----------------------------- | :---------------------------------------------------- |
| **1. Change**      | Change is inevitable.          | Budget for maintenance, not just development.         |
| **2. Complexity**  | Complexity always rises.       | Invest in **Preventive Maintenance** (Refactoring).   |
| **3. Regulation**  | Trends stay stable.            | Don't expect "miracle" spikes in productivity.        |
| **4. Stability**   | Work rate is constant.         | Adding more people won't necessarily speed things up. |

<summary>Types of Software Change and Rationale</summary>

Software change is categorized primarily by the **intent** behind the modification. The four standard categories (often called Swanson’s categories) are:

- **Corrective Change:**
  - **Description:** Reactive modification to repair faults (bugs) discovered after delivery.
  - **Rationale:** To restore the system to its intended functional state and ensure data accuracy.
- **Adaptive Change:**
  - **Description:** Adjusting the software to work in a new or changed environment (e.g., a new OS, cloud migration, or database upgrade).
  - **Rationale:** To ensure the software remains operational and compatible as external technology evolves.
- **Perfective Change:**
  - **Description:** Enhancing the software by adding new features, improving the user interface, or increasing performance.
  - **Rationale:** To meet evolving user needs, increase the system's value, and remain competitive.
- **Preventive Change (Refactoring):**
_ **Description:** Modifying software to improve maintainability or reliability without changing its external behavior.
_ **Rationale:** To reduce technical debt and lower the risk of future failures, making future changes easier and cheaper.

<details>

<summary>Importance of Distinguishing Change Types</summary>

Distinguishing between these changes is not just an academic exercise; it has significant practical implications for management:

1.  **Budgeting and Resource Allocation:** Management needs to know if they are spending money on **fixing the past** (Corrective) or **investing in the future** (Perfective/Preventive).
2.  **Performance Metrics:** High levels of corrective maintenance indicate poor initial development quality, whereas high perfective maintenance suggests a healthy, growing product.
3.  **Risk Management:** Preventive changes are low-risk improvements, while adaptive changes are high-priority necessities to avoid system obsolescence.
4.  **Strategic Planning:** Understanding the "Maintenance Ratio" helps organizations decide when a system has become too expensive to maintain and should be retired or replaced.

</details>

<details>

<summary>Is "Ongoing Support" Maintenance?</summary>

The view that support should be excluded from maintenance is a common point of debate. Here is an analysis of that perspective:

**The Argument for Exclusion**

- **Nature of the Task:** Support (Help Desks, User Training, Technical Consulting) often involves explaining how the system works rather than changing the code. If no code is changed, some argue it isn't "maintenance" in an engineering sense.
- **Budgeting:** Organizations often separate "Operations" (Support) from "Maintenance" (Engineering) to better track technical vs. human-centric costs.

**The Argument for Inclusion (The Modern View)**

- **The Holistic Definition:** Most industry standards (like ISO/IEC 14764) view maintenance as the total set of activities required to provide **post-delivery service**.
- **Root Cause Analysis:** Support is the "front line" that identifies the need for maintenance. A user support call often reveals a bug that requires a corrective change or a lack of usability that requires perfective change.
- **Feedback Loop:** Support provides the documentation and operating procedures mentioned in the standard definition of software. Updating a manual is a "modification of a software product" just as much as changing a line of code.

**Conclusion:** While support might not always result in code modification, it is an **integral part of the maintenance lifecycle**. Excluding it ignores the fact that software is a service, not just a static file. Effective maintenance is impossible without the insights gained through ongoing support.
