Chapter 5

Traditional Process Model​

The Traditional Process Model in software maintenance is essentially an extension of the linear, sequential development approach (similar to the waterfall model), applied to handling maintenance changes.

Code and Fix Model​

The simplest model:

  • Write code
  • Fix issues as they appear
  • No formal planning

Flow

Code → Test → Fix → Repeat

Example

A student builds a small calculator:

  • Writes code directly
  • Fixes errors when program crashes

Advantages

  • Very simple
  • Suitable for small projects

Disadvantages

  • No structure
  • Hard to maintain
  • Poor quality for large systems

Waterfall Model​

A sequential model where each phase is completed before moving to the next.

Phases:

  1. Requirements
  2. Design
  3. Implementation
  4. Testing
  5. Deployment
  6. Maintenance

Example

Library management system:

  • Requirements fixed at start
  • Developed step by step

Advantages

  • Easy to understand
  • Well-documented

Disadvantages

  • Not flexible
  • Changes are difficult

Spiral Model​

The Spiral Model is a risk-driven, iterative process model that combines:

  • Iteration (like iterative enhancement)
  • Systematic control (like traditional models)
  • Explicit risk analysis at every stage

Instead of a straight line, development proceeds in a spiral of repeated cycles

Each maintenance request becomes a spiral cycle:

  1. Define the change request
  2. Analyze risks (impact on legacy system)
  3. Implement carefully
  4. Validate with users
  5. Repeat for next change

Advantages

  • Better handling of uncertainty
  • Reduces chances of failure
  • Supports complex changes

Disadvantages

  • Complex to manage
  • Expensive
  • Not suitable for small changes

Iterative Model​

Develop software in repeated cycles, improving each version.

Example

E-learning platform:

  • Version 1: basic courses
  • Version 2: add quizzes
  • Version 3: add certificates

Advantages

  • Early working system
  • Easy to handle changes

Disadvantages

  • Requires planning
  • Can become complex

Agile Model​

A flexible model focusing on:

  • Continuous delivery
  • Customer feedback
  • Small iterations (sprints)

Example

Social media app:

  • Weekly updates
  • Continuous feature improvements

Advantages

  • Highly flexible
  • Fast delivery

Disadvantages

  • Less documentation
  • Needs experienced team

Maintainance Process Model​

Maintenance process models are structured approaches that define how software changes are handled after deployment.

Quick-Fix Model​

The quick-fix model is an informal maintenance approach where developers make direct changes to the code to solve a problem as quickly as possible.

Key features:

  • Immediate correction of faults
  • Minimal or no documentation
  • No formal analysis or design phase
  • Changes are often made directly in the codebase

Advantages:

  • Very fast response to urgent issues
  • Useful for emergency bug fixes

Disadvantages:

  • Leads to poor system structure over time
  • Difficult to maintain and understand code
  • Creates “technical debt”

Boehm’s Model (Spiral Model for Maintenance/Development)​

Barry Boehm introduced the Spiral Model, which is a risk-driven iterative process model.

Key features:

  • Development occurs in repeated cycles (spirals)
  • Each cycle includes:
    1. Planning
    2. Risk analysis
    3. Engineering (design & implementation)
    4. Evaluation by users
  • Strong focus on risk management

Advantages:

  • Good for large, complex, high-risk systems
  • Early identification and reduction of risks
  • Flexible and iterative

Disadvantages:

  • Expensive and complex to manage
  • Requires expertise in risk analysis

Osborne’s Maintenance Model​

Osborne’s model treats maintenance as a formal engineering process similar to development.

Key features:

  • Structured steps:
    1. Problem identification
    2. Impact analysis
    3. Change specification
    4. Design and implementation
    5. Testing
    6. Documentation update
  • Strong emphasis on control and documentation

Advantages:

  • Produces well-documented and maintainable systems
  • Reduces risk of introducing new errors
  • Structured and disciplined approach

Disadvantages:

  • Slower than quick-fix approach
  • May be heavy for small or urgent fixes

Iterative Model​

The iterative model develops software through repeated cycles, gradually improving the system.

Key features:

  • Software is built in small versions (iterations)
  • Each iteration includes:
    • Requirements
    • Design
    • Implementation
    • Testing
  • Feedback is used to improve the next version

Advantages:

  • Early delivery of partial working system
  • Flexible to changing requirements
  • Easier to find and fix errors

Disadvantages:

  • Requires good planning and management
  • Final system structure may degrade without control

Reuse-Oriented Model​

The reuse-oriented model builds systems by assembling existing components rather than developing everything from scratch.

Key features:

  • Uses reusable software components or services

  • Includes:

    • Component analysis
    • Requirements modification
    • System integration

  • Focus on reuse of existing software assets

Advantages:

  • Faster development time
  • Lower cost
  • Increased reliability (reused components are often tested)

Disadvantages:

  • Difficult to find perfectly matching components
  • Integration issues between components
  • Limited flexibility in design

When NOT to Make a Change​

Sometimes, not making a change is the better decision:

  • If the risk is higher than the benefit
  • If the system is stable and the change is minor
  • If resources are limited
  • If it may introduce instability

Process Maturity​

Process maturity refers to the degree of discipline, standardization, and optimization in how an organization performs software processes (especially maintenance).

In simple terms:

How well-defined, repeatable, and controlled the maintenance process is.

Levels of Process Maturity

Although Grubb discusses it in maintenance context, it aligns with general maturity ideas (similar to capability models):

  1. Initial
    • No formal process
    • Work done reactively
    • Heavy dependence on individual skill
  2. Repeatable level
    • Basic procedures exist
    • Some consistency in handling similar problems
    • Similar problems solved in similar ways
  3. Defined level
    • Standard processes are documented
    • Maintenance activities follow structured models
  4. Managed level
    • Processes are measured and monitored
  5. Optimizing level
    • Continuous improvement based on feedback

Software Experience Bases​

A software experience base is a structured repository of knowledge gained from past software projects and maintenance activities.

It stores “what we learned” so we don’t repeat mistakes.

What it Contains

A software experience base may include:

  1. Problem histories
    • Past bugs and failures
    • How they were fixed
  2. Change records
    • What changes were made
    • Why they were made
  3. Solutions and patterns
    • Reusable fixes
    • Standard approaches
  4. Metrics and data
    • Time taken for fixes
    • Cost of maintenance
    • Frequency of errors
  5. Lessons learned
    • What went wrong
    • What worked well

The main goal is knowledge reuse:

  • Avoid repeating past mistakes
  • Speed up maintenance decisions
  • Improve consistency

Question​

1. Define the terms process, life-cycle and model.

Process

A process is a structured set of activities, methods, and tasks used to achieve a specific goal. In software engineering, it refers to the steps followed to develop, test, deliver, and maintain software.

Life-cycle

A life-cycle is the complete sequence of stages that a system or product goes through from its beginning to its end. For software, this typically includes stages like requirements, design, implementation, testing, deployment, and maintenance.

Model

A model is a simplified representation of a real-world system or process. In software engineering, a model describes how the development life-cycle is organized and managed (e.g., Waterfall, Spiral, Agile).

2. Explain the differences between a software life-cycle and a software process

A software life-cycle and a software process are related but not the same:

Software Life-cycle

  • Focuses on the phases a software product goes through from start to finish.
  • Describes the timeline of development and maintenance.
  • Example phases: requirements → design → coding → testing → deployment → maintenance.
  • It is product-oriented (focused on the software itself).

Software Process

  • Focuses on the activities, methods, and practices used to build and maintain software.
  • Describes how work is done, not just the stages.
  • Includes things like coding standards, testing methods, project management techniques, etc.
  • It is activity-oriented (focused on how development is carried out).

Key Difference

  • The life-cycle tells you what stages software goes through.
  • The process tells you how those stages are performed.
3. Investigate the ways in which the basic waterfall model has been enhanced and modified. Describe these modifications and explain why they were made.

The basic Waterfall model is a linear, phase-by-phase approach (requirements → design → implementation → testing → maintenance). Because real projects are rarely so linear, several improvements were introduced.

1. Feedback (Iterative Waterfall)

  • What changed: Allows feedback loops between phases (e.g., testing may send issues back to design or requirements).
  • Why it was made: Real projects often discover errors late; strict one-way flow is unrealistic.

2. V-Model (Verification and Validation model)

  • What changed: Extends Waterfall by pairing each development phase with a corresponding testing phase.
    • Requirements ↔ Acceptance testing
    • Design ↔ System testing
    • Coding ↔ Unit testing
  • Why it was made: To improve early test planning and ensure quality at every stage.

3. Incremental Model

  • What changed: System is developed in small functional parts (increments) instead of all at once.
  • Why it was made: To deliver working software earlier and reduce risk.

4. Iterative Model

  • What changed: Software is built in repeated cycles, refining the system each time.
  • Why it was made: Requirements change over time; iteration allows adaptation.

5. Spiral Model

  • What changed: Combines iteration with risk analysis at every cycle.
  • Why it was made: To manage high-risk, large projects more effectively.

Summary

These modifications were introduced mainly to address:

  • Rigid structure of Waterfall
  • Late detection of errors
  • Changing requirements
  • Need for faster delivery and risk control
4. Compare and contrast Osborne's maintenance process model with the other maintenance process models dealt with in this chapter.

Osborne’s Maintenance Process Model

  • Focuses on maintenance as a structured engineering activity
  • Emphasizes:
    • Problem identification
    • Impact analysis
    • Modification design
    • Implementation
    • Testing
    • Documentation update

It treats maintenance like a mini software development lifecycle.

Comparison with other maintenance models

1. Quick-Fix Model

  • Approach: Make immediate code changes to solve the problem.
  • Comparison:
    • Osborne is structured and planned
    • Quick-fix is ad hoc and code-focused
  • Result: Quick-fix leads to technical debt; Osborne reduces it.

2. Reengineering Model

  • Approach: Improve structure of existing system (refactoring, restructuring).
  • Comparison:
    • Osborne focuses on corrective and controlled change
    • Reengineering focuses on improving internal architecture
  • Result: Reengineering is broader and more transformative.

3. Evolutionary Maintenance Model

  • Approach: System evolves continuously with changes integrated into design.

  • Comparison:

    • Osborne is more process-heavy and formal
    • Evolutionary model is more adaptive and ongoing

Key conclusion

Osborne’s model is more controlled and disciplined, while others vary between:

  • fast but messy (quick-fix)
  • structural improvement (reengineering)
  • continuous adaptation (evolutionary)
5. Describe how the ACME Health Clinic system might have been more effectively modified. Assume the same tight deadlines but investigate the incorporation of the quick-fix model into another, more structured model.

The quick-fix model is useful under tight deadlines but often causes poor long-term code quality. For the ACME Health Clinic system, a better approach would be to combine quick fixes with a structured maintenance model.

Proposed Hybrid Approach: “Controlled Quick-Fix within Structured Maintenance”

Step 1: Emergency Fix Layer (Quick-Fix)

  • Apply quick fixes only for:

    • Critical system failures
    • Patient safety issues
    • System downtime problems

  • Goal: restore system immediately

Step 2: Formal Logging System

  • Every quick fix must be recorded:

    • What was changed
    • Why it was changed
    • Possible side effects

  • Prevents loss of traceability

Step 3: Osborne-style Structured Maintenance Cycle

After stabilizing:

  • Perform:

    1. Impact analysis
    2. Proper redesign (if needed)
    3. Code refactoring
    4. Full testing
    5. Documentation update

Step 4: Scheduled Refactoring Phase

  • Allocate time after deadlines to:

    • Replace temporary fixes
    • Improve system structure
    • Remove duplicated or unsafe code

Why this approach works

  • Maintains system stability under tight deadlines
  • Prevents long-term degradation from quick fixes
  • Ensures medical system reliability and safety
  • Balances speed + quality

Final conclusion

Instead of relying purely on quick fixes, ACME should:

Use quick fixes only as emergency patches, then enforce a structured maintenance process (like Osborne’s model) to properly integrate and improve changes.

6. Software project reflection

What was the last software project you worked on? Was it commercial, undergraduate, or personal?

Write a critical appraisal of the life-cycle model used:

  • Was it well structured or ad hoc?
  • Would you choose a different model if restarting?

The last software project I worked on was a personal project: a web-based task management application with features like user authentication, task categorization, and deadline tracking.

Life-Cycle Model Used

The project loosely followed an Iterative Enhancement Model, but in practice it leaned toward a semi ad hoc approach.

Critical Appraisal

  1. Structure: Well-structured or Ad Hoc?

    • The project began with some planning and feature outlining.
    • Development proceeded in iterations (adding features step-by-step).
    • However:
      • Documentation was minimal
      • Design decisions were often made on the fly
      • Some quick fixes were applied without proper refactoring

    Conclusion: The process was partially structured but largely informal, especially in later stages.

  2. Strengths

    • Flexibility in adding new features
    • Fast development cycles
    • Ability to respond quickly to issues

  3. Weaknesses

    • Accumulation of technical debt
    • Inconsistent code structure
    • Lack of proper testing and documentation

Would I Choose a Different Model?

Yes.

If restarting, I would adopt a more disciplined Iterative Enhancement Model with elements of the Spiral Model:

  • Clear iteration planning
  • Regular refactoring
  • Risk analysis for major features
  • Better documentation and testing

This would improve maintainability and long-term scalability.

7. Library system failure scenario

You are the IT manager of a large library system that fails unexpectedly.

How would you respond:

  1. If the system must be restored within two hours?

  2. If the library can operate for several days without it?

  3. If the system must be restored within two hours

Approach: Quick-Fix / Emergency Response

Immediate Actions:

  • Identify the root cause quickly (hardware, software, network, database)
  • Switch to backup systems if available
  • Restart servers or services
  • Apply temporary patches or rollbacks

Priority:

  • Restore basic functionality, not perfection

Techniques:

  • Use system logs for rapid diagnosis
  • Roll back to the last stable version
  • Disable non-critical features

Communication:

  • Inform staff about limited functionality
  • Provide temporary manual procedures if needed

Goal: Fast recovery with minimal downtime, even if the solution is temporary.

  1. If the library can operate for several days without it

Approach: Structured / Iterative or Spiral Model

Step-by-Step Actions:

  1. Problem Analysis
    • Conduct detailed root cause analysis
    • Assess system-wide impact
  2. Planning
    • Define recovery strategy
    • Allocate resources and timeline
  3. Design
    • Redesign faulty components if necessary
    • Improve system architecture
  4. Implementation
    • Fix underlying issues
    • Refactor problematic modules
  5. Testing
    • Perform thorough testing:
      • Unit testing
      • Integration testing
      • Regression testing
  6. Deployment
    • Gradual rollout
    • Monitor system performance

Key Difference Between the Two Scenarios

Aspect2-Hour RecoverySeveral Days Available
ApproachQuick-fixStructured/iterative
FocusSpeedQuality & stability
Risk HandlingMinimalThorough
OutcomeTemporary solutionLong-term solution
