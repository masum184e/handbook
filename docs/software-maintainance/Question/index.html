Semester Final​

Question 01​

(a) Compare and contrast new development and software maintenance in terms of their purpose, scope, and challenges. Why is maintenance considered an ongoing process, while new development follows a structured timeline?

Software Development

Purpose:

  • To create a new software system from scratch to meet specific requirements.

Scope:

  • Includes requirement analysis, system design, coding, testing, and deployment.

Challenges:

  • Understanding user requirements
  • Designing scalable architecture
  • Managing project timelines and resources
  • Handling evolving requirements during development

Software Maintenance

Purpose:

  • To modify and improve existing software after deployment.

Scope:

  • Bug fixing
  • Performance improvements
  • Security updates
  • Adding new features or adapting to new environments

Challenges:

  • Understanding legacy code
  • Lack of documentation
  • Risk of introducing new bugs
  • Compatibility issues with new technologies

Why Maintenance is Ongoing

Software maintenance is continuous because:

  • User needs change over time
  • Technology evolves
  • Security vulnerabilities emerge
  • Operating systems and hardware are updated
(b) Discuss the impact of software maintenance on the long-term sustainability of a system. How do maintenance activities contribute to the security, performance, and usability of software over time?

Software maintenance ensures long-term sustainability by keeping software functional, secure, and efficient.

Security

  • Fixes vulnerabilities and security flaws
  • Applies security patches
  • Protects systems from cyber threats

Performance

  • Optimizes code and database queries
  • Improves system response time
  • Reduces memory and CPU usage

Usability

  • Updates the user interface
  • Fixes user experience issues
  • Adds features based on user feedback

Overall Impact

Maintenance increases system lifespan, reduces system failures, and ensures the software remains relevant to changing business and technological environments.

(c) Given that maintenance often involves working with legacy systems, what strategies can software engineers use to efficiently manage and update older software without introducing new issues?

Strategies include:

  1. Refactoring: Improve code structure without changing functionality. It makes legacy code easier to maintain.
  2. Comprehensive Testing: Use automated tests to verify system stability. It prevents new bugs during updates
  3. Documentation Improvement: Update missing or outdated documentation. It helps developers understand the system
  4. Modular Updates: Modify small components instead of the whole system. It reduces risk of system-wide failures
  5. Version Control Systems: Track changes and enable rollback if issues occur

Question 02​

(a) Explain the role of Change Management and Configuration Management in software maintenance. How do these components help maintain software quality and stability?

Change Management

Change management controls how modifications are requested, evaluated, approved, and implemented.

Roles:

  • Tracks change requests
  • Evaluates impact before implementation
  • Ensures proper approval before changes

Benefits:

  • Prevents unauthorized modifications
  • Reduces system disruption
  • Maintains software reliability

Configuration Management

Configuration management manages software components, versions, and system settings.

Roles:

  • Maintains version control
  • Tracks system configurations
  • Ensures consistency across environments

Benefits:

  • Avoids version conflicts
  • Helps developers revert to stable versions
  • Ensures system stability during updates

Together, these processes ensure that software changes are controlled, documented, and safely implemented.

(b) Why is Testing and Quality Assurance essential in the maintenance phase? Describe the types of testing involved in software maintenance and their importance.

Testing ensures that updates or fixes do not break existing functionality.

Importance of Testing in Maintenance

  • Detects new bugs introduced during updates
  • Ensures system reliability
  • Maintains software quality

Types of Testing

1. Regression Testing

  • Ensures existing features still work after modifications.

2. Unit Testing

  • Tests individual components or functions.

3. Integration Testing

  • Ensures updated modules work with other components.

4. System Testing

  • Verifies the whole system functions correctly.

5. Acceptance Testing

  • Confirms the system meets user requirements after updates.

These tests ensure the system remains stable and reliable after maintenance activities.

(c) Discuss the significance of Performance Monitoring and Optimization in maintaining software efficiency. What tools or techniques can be used to ensure software performance remains optimal over time?

Significance

Performance monitoring helps identify system bottlenecks and inefficiencies.

Benefits:

  • Prevents slow system response
  • Detects resource overuse
  • Improves scalability and reliability

Techniques

  • Code optimization
  • Database query optimization
  • Load balancing
  • Caching frequently used data

Tools

Common tools used for performance monitoring include:

  • Application Performance Monitoring (APM) tools
  • Log analysis tools
  • System monitoring dashboards
  • Automated alert systems

These tools help detect performance issues early and maintain efficient software operation.

Question: 03​

(a) Discuss different types of software change and give reasons why changes occur.

Types of Software Changes

  1. Corrective Maintenance: Fixes bugs and errors found after deployment.
  2. Adaptive Maintenance: Adjusts software to new environments such as new hardware or operating systems.
  3. Perfective Maintenance: Improves performance or adds new features.
  4. Preventive Maintenance: Improves system maintainability to prevent future problems.

Reasons Changes Occur

  • Discovery of bugs
  • Changing user requirements
  • Security updates
  • Technology upgrades
  • Regulatory or business changes
(b) How can AI and machine learning improve predictive software maintenance? Discuss the potential benefits and challenges of integrating AI-driven analytics into maintenance workflows.

Artificial Intelligence and Machine Learning analyze system data to predict future failures.

Examples:

  • Detect abnormal system behavior
  • Predict software failures
  • Recommend maintenance actions

Benefits

  • Early detection of issues
  • Reduced downtime
  • Improved system reliability
  • Better resource planning

Challenges

  • High implementation cost
  • Need for large datasets
  • Complexity in integrating AI with existing systems
  • Data privacy and security concerns

Despite challenges, AI can significantly improve maintenance efficiency and decision-making.

(c) Why is software documentation considered a critical component of maintenance? What are the challenges in maintaining up-to-date documentation, and how can organizations ensure its effectiveness?

Software documentation helps developers understand system design, features, and functionality.

Benefits:

  • Easier debugging and updates
  • Faster onboarding of new developers
  • Clear understanding of system architecture

Challenges

  • Documentation becomes outdated quickly
  • Developers may neglect documentation updates
  • Lack of standard documentation practices

Ensuring Effective Documentation

  1. Continuous Documentation Updates: Update documentation during every code change.
  2. Documentation Standards: Use consistent templates and formats.
  3. Automated Documentation Tools: Generate documentation from code comments.
  4. Documentation Reviews: Include documentation checks during code reviews.

This ensures that documentation remains accurate and useful for long-term maintenance.

Question: 09​

(a) Challenges in Modernizing a 30-Year-Old Legacy System

Updating a three-decade-old inventory system while keeping it operational (often called "changing the engines while the plane is flying") presents several critical challenges:

  • Legacy Knowledge Gap: After 30 years, the original developers are likely gone. Documentation may be missing or outdated, making it difficult to understand the business logic embedded in the old code.
  • Data Integrity and Migration: Moving 30 years of data to a modern architecture is risky. Ensuring that data remains consistent and uncorrupted during the transition is a major hurdle.
  • Zero-Downtime Requirement: Since the system must remain operational across multiple warehouses, you cannot simply "turn it off" to upgrade. This requires a phased approach (like the Strangler Fig pattern) which increases architectural complexity.
  • Integration with Modern Tech: The old system may use obsolete protocols or flat-file databases that do not easily communicate with modern APIs or cloud environments.
  • Resistance to Change: Staff accustomed to the 30-year-old interface may resist the new system, requiring significant training and change management.

(c) Why Making the Product Open Source Could Be Beneficial

In the scenario described, open-sourcing the specialized product offers several strategic advantages:

  • Co-Development with Customers: Since the new customer wants involvement in the configuration, open-sourcing allows them (and their developers) to contribute code directly. This reduces the workload on the small company while meeting specific requirements.
  • Scalability: Doubling the customer base can overwhelm a "small company." An open-source model allows the community to help identify bugs and provide support, making the software more robust as it grows.
  • Standardization: If the product becomes an open-source standard in its niche, it becomes harder for competitors to displace it. The company can then pivot its business model toward consulting and high-end support rather than just selling licenses.
  • Trust and Transparency: New, large customers may feel more secure knowing they have access to the source code, ensuring they aren't "locked in" if the small provider goes out of business.
