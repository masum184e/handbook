Chapter 7

Reverse Engineering (in Software Maintenance)​

Reverse engineering is:

The process of analyzing an existing software system to identify its components, relationships, and structure, and to represent it at a higher level of abstraction.

  • It works backward from implementation (code) → design → requirements.
  • Usually starts from source code, because documentation is often missing or outdated.

Purpose of Reverse Engineering​

According to the book, reverse engineering is mainly used to:

  • Improve program understanding
  • Recover lost or outdated documentation
  • Identify system components and dependencies
  • Support maintenance and modification
  • Extract reusable components
  • Reduce maintenance effort and complexity

Reverse engineering itself does not modify the system—it only helps understand it.

Abstraction in Reverse Engineering​

Reverse engineering relies heavily on abstraction, which means:

  • Focusing on important details
  • Ignoring unnecessary low-level details

Types of abstraction:​

  1. Function abstraction
    • Focuses on data structures
    • Ignores implementation details
    • Example: Treating a stack as push/pop instead of array implementation
  2. Data abstraction
    • Focuses on what a function does, not how
    • Identify data structures
    • Example: sort() instead of sorting algorithm details
  3. Process abstraction
    • Identify control flow and execution order
    • Example: loops, conditionals, program sequence

Levels of Reverse Engineering​

The book explains reverse engineering in terms of levels of abstraction. These levels show how far we move from low-level code to high-level understanding.

Redocumentation (Lowest Level)​

  • Creating documentation at the same level of abstraction as the existing system.

Characteristics

  • Does not change abstraction level
  • Focuses on improving readability and documentation
  • Produces:
    • Comments
    • Flowcharts
    • Code documentation

Purpose: Help developers understand poorly documented code

Example: Generating UML diagrams from code without changing design understanding

Design Recovery (Intermediate Level)​

Recovering the design structure of the system from code.

Characteristics

  • Moves from implementation → design level
  • Uses:
    • Domain knowledge
    • Heuristics
    • External information

What it produces

  • Architecture diagrams
  • Module structure
  • Relationships between components

Key Insight: Not just extracting information—interpreting it

Specification Recovery (Highest Level)​

Recovering high-level requirements and specifications of the system.

Characteristics

  • Moves from code → requirements
  • Highest level of abstraction
  • Answers:
    • What does the system do?
    • Why does it exist?

Output

  • Functional specifications
  • Requirement models

Summary of Levels​

LevelAbstractionFocusOutput
RedocumentationSame levelCode understandingComments, diagrams
Design RecoveryHigherSystem designArchitecture, modules
Specification RecoveryHighestSystem purposeRequirements

Important Concept​

  • Reverse engineering is a bottom-up process:
    • Start with code (low level)
    • Gradually move to design and specification
  • If abstraction does not increase, it is: Redocumentation
  • If abstraction increases, it becomes: Design or Specification Recovery

Relationship with Other Techniques​

Reverse engineering supports:

  • Forward Engineering → building new systems
  • Restructuring → improving code structure
  • Reengineering → transforming the system completely

Restructuring​

Transforming a system to improve its structure without changing its external behavior

Types of Restructuring​

  1. Code Restructuring
    • Improves code readability
    • Example: Removing goto statements, Simplifying conditions
  2. Data Restructuring
    • Improves data organization
    • Example: Converting file structures to databases
    • Key Characteristics
      • Behavior remains unchanged
      • Focus is on internal quality
      • Often automated

Reengineering​

The process of examining and altering a system to reconstitute it in a new form

Reengineering combines:

  • Reverse engineering (understanding)
  • Forward engineering (rebuilding)

Activities

  1. Reverse engineering → understand system
  2. Restructuring → clean/improve
  3. Forward engineering → rebuild

Goals

  • Improve maintainability
  • Modernize systems
  • Increase performance

Forward Engineering​

The traditional process of moving from high-level abstraction to implementation

Direction: Requirements → Design → Code

Characteristics

  • Builds new systems
    • Uses:
    • Specifications
    • Design models
  • Opposite of reverse engineering

Question​

Discuss the application of reverse engineering techniques to maintenance problems.

Reverse engineering plays a critical role in software maintenance, especially for legacy systems.

  1. Program Understanding
    • Problem: Legacy systems lack documentation
    • Solution: Reverse engineering extracts: Structure, Logic, Dependencies
    • Helps maintainers understand the system before making changes
  2. Impact Analysis
    • Problem: Hard to predict effects of changes
    • Solution: Identify relationships between components
    • Helps answer: What will break if I change this?
  3. Redocumentation
    • Problem: Missing or outdated documentation
    • Solution: Generate:, Diagrams, Reports, Comments
  4. Design Recovery
    • Problem: Original design lost over time
    • Solution: Reconstruct architecture and module structure
  5. Migration and Reengineering
    • Problem: Old systems need modernization
    • Solution: Reverse engineering helps: Understand old system, Prepare for transformation
  6. Debugging and Error Correction
    • Problem: Difficult to locate bugs
    • Solution: Analyze control and data flow, Trace execution paths
  7. Reuse of Components
    • Problem: Reinventing existing solutions
    • Solution: Identify reusable modules
  8. Security Analysis - Problem: Unknown vulnerabilities - Solution: Analyze code behavior and structure
Discuss reverse engineering tools and identify some of their limitations.

Reverse engineering tools are software tools that help analyze an existing system and extract higher-level representations (design, structure, or documentation) from source code or binaries.

  1. Code Analysis Tools: Analyze source code structure Provide: Control flow, Data flow, Call graphs
  2. Redocumentation Tools: Generate documentation from code
  3. Design Recovery Tools: Extract higher-level design information
  4. Data Reverse Engineering Tools: Focus on data structures and databases

Limitations

  1. Limited Understanding of Semantics
  2. Incomplete Design Recovery
  3. Difficulty with Poor Code Quality
  4. Scalability Issues
  5. Limited Automation
  6. Language and Platform Dependency
  7. Loss of High-Level Information
2. Carry out specification and design recovery on all or parts of a software system that you are not familiar with (the system should be at least 2K lines of code in length). What technique(s) do you use to identify the specification and design and why? What form of representation do you consider suitable for these tasks? Indicate your reasons. What lessons did you

For this exercise, I selected a medium-sized open-source system (~2K–3K LOC):

A Library Management System (LMS) written in Java/Python (Modules: user management, book catalog, issue/return, fine calculation)

Specification and Design Recovery

  1. Specification Recovery

    To recover the system specification (what the system does):

    • Read:
      • Main entry files (main(), controllers)
        • User interface code (menus, forms)
    • Analyze:
      • Input/output behavior
      • User interactions
    • Observe:
      • System responses to actions

  2. Recovered Specification (Example)
    Functional Requirements:

    • Users can:
      • Search books
      • Borrow and return books
    • Admin can:
      • Add/remove books
      • Manage users
    • System:
      • Tracks due dates
      • Calculates fines

    Non-functional Observations:

    • Console-based interface
    • File/database storage
    • Basic validation

  3. Design Recovery
    To recover system design (how it works):

    • Identify:
      • Modules and classes
      • Relationships between components
    • Analyze:
      • Function calls
      • Data flow
      • Control flow

Recovered Design (Example)

Modules Identified:

  • UserModule
  • BookModule
  • TransactionModule
  • Database/FileHandler

Relationships:

  • User → interacts with → Transaction
  • Transaction → accesses → Book + Database

Techniques Used

  1. Static Analysis

    • Reading source code without execution
    • Identified:
      • Class structure
      • Function definitions
      • Dependencies

    Why used?

    • Safe and straightforward
    • Good for structural understanding

  2. Control Flow Analysis

    • Tracing execution paths (if-else, loops)
    • Why used? Helps understand logic and decision-making

  3. Data Flow Analysis

    • Tracking how data moves through the system

    Why used?

    • Essential for understanding:
      • Inputs/outputs
      • Data dependencies

  4. Dynamic Analysis (Optional but useful)

    • Running the system and observing behavior

    Why used?

    • Confirms assumptions from static analysis
    • Helps understand runtime behavior

  5. Abstraction

    • Ignoring low-level code details
    • Focusing on:
      • Functions
      • Modules
      • System goals

    Why used? Necessary to move from code → design → specification

3. A bank has a substantial investment in a Cobol software system that is at least one million lines of code in length and has been running for over 20 years. It is used on a daily basis to perform various operations such as managing customer accounts and loans. After several years of modification - both planned and ad hoc - the system has become too expensive to maintain. As a result, the bank wants some advice on the next step to take. Suppose that you have been employed as a software maintenance consultant. What advice would you give the bank? Indicate the reasons for any recommendations you make.

Here’s the advice I would give as a software maintenance consultant.

  1. First Step: Understand Before Acting

    Recommendation: Perform Reverse Engineering + System Assessment

    Before any decision:

    • Recover system architecture, modules, and dependencies
    • Identify:
      • Critical business functions (accounts, loans, transactions)
      • High-maintenance/problem areas
      • Obsolete components

    Reason:

    • After 20 years, documentation is likely outdated or missing
    • Decisions without understanding will increase risk and cost

  2. Evaluate Strategic Options

    The bank has four main options:

    • Option 1: Continue Maintenance (Not Recommended Alone): Keep fixing and modifying the existing COBOL system

      Problems

      • Maintenance cost will keep increasing
      • System complexity will worsen
      • Skills (COBOL developers) are becoming scarce

      Conclusion: Only a short-term solution

    • Option 2: Complete System Replacement (High Risk): Rewrite the entire system using modern technology

      Problems

      • Extremely expensive and time-consuming
      • High failure rate in large system rewrites
      • Risk of losing critical business logic

      Conclusion: Avoid “big bang” replacement

    • Option 3: Reengineering (Recommended Core Strategy): Improve and transform the existing system

      Activities

      • Reverse engineering → understand system
      • Restructuring → improve code and data
      • Partial redesign → modernize components

      Benefits

      • Preserves proven business logic
      • Reduces risk compared to full rewrite
      • Improves maintainability

      Reason: The system has worked reliably for 20 years—this knowledge is valuable

    • Option 4: Incremental Modernization (Best Practical Approach): Gradually replace or upgrade parts of the system

      Approach

      • Identify modules:
        • High-change → replace first
        • Stable → keep or wrap

      Techniques

      • Introduce APIs around COBOL system
      • Migrate modules to modern platforms step-by-step

      Reason:

      • Minimizes disruption to daily banking operations
      • Allows continuous system use during transition

The bank should adopt a reengineering-based incremental modernization strategy, combining reverse engineering, restructuring, and gradual replacement to reduce risk, preserve business logic, and improve maintainability.
